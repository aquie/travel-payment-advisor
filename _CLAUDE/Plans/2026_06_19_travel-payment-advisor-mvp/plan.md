# Travel Payment Advisor MVP 구현 계획

## 1. 목적과 성공 기준

- 2026년 7월 1일 이후 일본에서 JPY로 결제할 때 Travel Wallet, Toss Bank 체크카드, Naver Pay 해외 QR, 아시아나 신한카드 Air 1.5의 예상 유효 원화 비용을 비교한다.
- 사용자가 현장에서 결제금액과 최근 환율만 빠르게 입력하면 추천 수단, 차선 수단, 예상 차이와 계산 근거를 확인할 수 있어야 한다.
- 이 앱은 정산 또는 가계부가 아니라 의사결정 보조 도구다. 실제 청구액과 달라질 수 있음을 명시하고, 정확한 상품별 환율을 확보하지 못한 경우 추정 API 값을 대신 사용하지 않는다.
- 모바일에서 설치·오프라인 실행 가능한 정적 PWA로 제공하며 개인 결제 데이터와 설정은 기기 밖으로 전송하지 않는다.

## 2. MVP 범위와 제외 범위

### 포함

- 일본/JPY 한 국가·통화
- 4개 결제수단 비교와 세부 비용 내역
- 공통 환율 입력 및 Naver Pay·신한카드의 선택적 환율 보정
- Naver Pay 잔여 캐시백과 신한카드 전월 실적·월간 해외 이용액의 러프한 반영
- 최근 7일 추천 결과, 마지막 입력값, 설정의 로컬 저장
- 전체 데이터 내보내기·삭제
- PWA 설치, 최근 입력 환율을 이용한 오프라인 계산
- iPhone 크기 중심의 반응형 UI와 접근성 기본 검증

### 제외

- 환율 API 자동 호출, 서버, 사용자 계정, 클라우드 동기화
- 실제 카드 승인·매입 내역 연동과 원 단위 정산 보장
- 결제 기록에 따른 캐시백·마일리지 한도의 자동 차감
- 일본 외 국가, ATM 인출, DCC, 할부, 취소·부분취소
- Naver Pay 머니카드 및 2026년 6월 30일 이후 공식 연장이 확인되지 않은 PayPay 별도 즉시 할인

## 3. 기술 방향

- Vite + React + TypeScript를 사용한다.
- 계산 엔진은 React와 저장소에 의존하지 않는 순수 TypeScript 모듈로 작성하고 Vitest로 단위 테스트한다.
- 스타일은 별도 UI 프레임워크 없이 CSS 변수와 작은 컴포넌트 집합으로 구성한다. 초기 번들 및 유지보수 부담을 줄인다.
- `vite-plugin-pwa`로 manifest와 서비스 워커를 생성하고 앱 셸을 캐시한다.
- 로컬 데이터 규모가 작으므로 `localStorage`를 사용한다. 저장 데이터는 버전 필드를 가진 단일 JSON 문서로 관리한다.
- GitHub Pages를 기본 배포 대상으로 삼고 Vite의 base path를 저장소 이름에 맞게 설정한다.
- 비밀키는 브라우저 코드나 저장소에 두지 않는다. 공식 환율 자동 채우기가 필요해질 때 Cloudflare Pages의 서버 함수와 별도 검토를 거친다.

## 4. 사용자 흐름과 화면

1. 첫 화면에서 결제 예정일(기본값: 오늘), JPY 금액, 공통 환율을 입력한다.
2. 환율은 `100 JPY = n KRW` 형식으로 입력하며 마지막 입력값과 입력 시각을 함께 표시한다.
3. 환율 입력 후 24시간이 지나면 오래된 환율 경고를 표시하되 계산은 허용한다.
4. 고급 설정에서 USD/KRW, Naver Pay·신한카드 전용 환율 보정값과 혜택 상태를 수정한다.
5. 입력이 유효하면 네 수단의 예상 유효비용을 계산해 가장 낮은 순서로 표시한다.
6. 결과 상단에는 추천, 차선, 원화 차이를 표시한다. 1위와 2위의 유효비용 차이가 1위 비용의 1% 이하면 `차이가 작으므로 편의성·가맹점 수용 여부를 우선해도 됨`을 표시한다.
7. 각 결과를 펼치면 환산 원금, 퍼센트 수수료, 고정 수수료, 캐시백, 마일리지 가치와 적용 가정을 확인할 수 있다.
8. 유효한 계산 결과는 자동으로 최근 기록에 저장하고 7일이 지난 기록은 앱 시작과 신규 저장 시 삭제한다.
9. 설정 화면에서 JSON 내보내기와 전체 초기화를 제공한다. 초기화는 확인 단계를 거친다.

## 5. 입력과 공개 타입

구현 시 아래 의미를 유지하는 타입을 정의한다. 이름은 저장소 컨벤션에 맞춰 조정할 수 있지만 단위와 선택 여부는 바꾸지 않는다.

```ts
type QuoteInput = {
  quoteDate: string;                // YYYY-MM-DD
  purchaseAmountJpy: number;
  commonKrwPer100Jpy: number;
  usdKrw: number;                   // Toss의 USD 0.50 환산용
  naverKrwPer100Jpy?: number;       // 미입력 시 공통 환율
  shinhanKrwPer100Jpy?: number;     // 미입력 시 공통 환율
  naverEventApplied: boolean;
  naverCashbackRemainingKrw: number;
  shinhanPreviousMonthEligible: boolean;
  shinhanOverseasSpendThisMonthKrw: number;
  asianaMileValueKrw: 10 | 15 | 20;
};

type MethodBreakdown = {
  methodId: 'travel-wallet' | 'toss-bank' | 'naver-pay-qr' | 'shinhan-air-1.5';
  convertedKrw: number;
  percentageFeesKrw: number;
  fixedFeesKrw: number;
  cashbackKrw: number;
  earnedMiles: number;
  mileageValueKrw: number;
  effectiveCostKrw: number;
  assumptions: string[];
  warnings: string[];
};

type ComparisonResult = {
  calculatedAt: string;
  input: QuoteInput;
  rankedMethods: MethodBreakdown[];
  isCloseCall: boolean;
};
```

- 모든 금액 입력은 유한한 0 이상 숫자여야 하며 구매금액과 환율은 0보다 커야 한다.
- 계산 내부는 소수 정밀도를 유지하고 화면의 KRW 합계와 차이는 최종 단계에서 가장 가까운 원으로 반올림한다.
- 마일은 비교 목적의 근삿값으로 소수를 허용하고 화면에서는 소수 첫째 자리까지 표시한다.
- 동률은 반올림 전 유효비용, 그다음 현금성 비용(`converted + fees - cashback`), 마지막으로 고정된 수단 순서로 정렬한다.

## 6. 계산 규칙

공통식은 다음과 같다.

```text
effectiveCost = convertedKRW
              + percentageFees
              + fixedFees
              - cashback
              - mileageValue
```

### Travel Wallet

- `convertedKRW = JPY × commonKrwPer100Jpy / 100`
- JPY는 공식 약관상 매매기준율과 동일한 환율, 카드 결제 수수료는 0%로 취급한다.
- 현금성 기준선이므로 캐시백과 마일리지는 없다.

### Toss Bank

- `convertedKRW = JPY × commonKrwPer100Jpy / 100`
- 국제브랜드 수수료는 환산 원금의 1%로 근사한다.
- 해외서비스 수수료는 `USD 0.50 × usdKrw`로 근사한다.
- 일반 해외 캐시백은 환산 원금의 2%로 근사한다.
- 공식 페이지에서 일본 3% 이벤트 종료일 또는 연장이 명시적으로 재확인되지 않는 한 2026년 7월 1일 이후에는 3%를 적용하지 않는다.
- 실제 공식 캐시백은 USD 거래금액과 승인·매입 시점 환율을 사용하므로 결과에 근사 계산 경고를 붙인다.

### Naver Pay 해외 QR

- `convertedKRW = JPY × (naver override 또는 common rate) / 100`
- 이벤트 신청 여부가 참일 때만 `min(convertedKRW × 10%, remainingCashbackKrw)`를 캐시백으로 적용한다.
- 사용자는 보험 비교 여부 자체가 아니라 현재 남은 캐시백 금액을 러프하게 입력한다. 기본 잔여 한도는 일반 5,000원이며 보험 비교 완료 시 사용자가 10,000원으로 바꿀 수 있게 빠른 설정을 제공한다.
- 공식 이벤트 기간인 2026-01-01~2026-12-31 밖에서는 캐시백을 0으로 처리하고 기간 경고를 표시한다.
- 실제 원화 청구에 포함될 수 있는 네트워크·중개 수수료를 별도 추정하지 않는다. 사용자가 확인한 예상 환율을 보정값으로 넣을 수 있게 한다.

### 아시아나 신한카드 Air 1.5

- `convertedKRW = JPY × (shinhan override 또는 common rate) / 100`
- Mastercard 1%와 신한 해외서비스 수수료 0.18%를 각각 환산 원금 기준으로 근사한다.
- 전월 50만원 조건 미충족 시 마일은 0이다.
- 조건 충족 시 기본 마일은 `convertedKRW / 1,000 × 1.5`로 근사하고 월 한도는 없다.
- 추가 마일의 기존 사용량은 `min(2,000, monthlyOverseasSpend / 1,000 × 1.5)`로 근사한다.
- 이번 거래의 추가 마일은 `min(convertedKRW / 1,000 × 1.5, 2,000 - estimatedUsedAdditionalMiles)`이다.
- `mileageValue = (baseMiles + additionalMiles) × selectedMileValue`이며 기본 프리셋은 15원/마일이다.
- 실제 Mastercard JPY→USD 변환과 신한 접수일 전신환매도율을 재현하지 않으며, 신한 전용 환율 보정 입력과 경고로 불확실성을 드러낸다.

## 7. 날짜 제한 규칙과 공식자료 관리

- 수수료·혜택은 UI 조건문에 흩어놓지 않고 `validFrom`, `validThrough`, 출처 URL, 마지막 확인일을 가진 데이터 규칙으로 관리한다.
- 최초 규칙의 공식 확인 기준일은 2026-06-19로 기록한다.
- 구현 시작 시와 배포 직전에 다음 공식 페이지를 다시 확인한다.
  - Travel Wallet 트래블페이 수수료 약관
  - Toss Bank 체크카드 혜택 페이지
  - Naver Pay 2026 해외여행 10% 페이백 이벤트 및 환율 안내
  - 아시아나 신한카드 Air 1.5 혜택과 신한카드 해외이용 수수료 안내
- 확인되지 않은 이벤트 연장이나 환율 산식을 추정해 활성화하지 않는다. 날짜가 지난 규칙은 자동 적용하지 않고 사용자에게 공식 조건 확인 경고를 보여준다.

## 8. 로컬 저장과 개인정보

- 저장 문서에는 `schemaVersion`, 마지막 입력, 사용자 설정, 최근 추천 결과만 포함한다.
- 추천 기록에는 계산 시각, 핵심 입력, 순위와 비용 내역을 저장하며 실제 결제 여부나 상점명은 받지 않는다.
- 기록은 최대 7일만 보관하고 오래된 항목을 자동 정리한다.
- 내보내기는 저장 문서 전체를 JSON 파일로 제공한다.
- 전체 초기화는 입력·설정·기록과 캐시 가능한 사용자 데이터까지 삭제하되 PWA 앱 셸은 유지한다.
- 저장 문서 파싱 실패나 알 수 없는 버전은 앱 중단 대신 안전한 기본값으로 복구하고 초기화 안내를 표시한다.

## 9. 테스트 계획

### 계산 엔진 단위 테스트

- Travel Wallet이 공통 환율에 따른 기준 비용과 정확히 일치한다.
- Toss의 1% 수수료, USD 0.50 고정 수수료, 2% 캐시백이 각각 분리되어 계산된다.
- Toss의 소액·고액 거래에서 고정 수수료 때문에 추천 순위가 달라지는 경계 사례를 검증한다.
- Naver 이벤트 미신청·기간 전·기간 후에는 캐시백이 0이다.
- Naver 캐시백은 잔여액을 초과하지 않으며 0원 잔여 한도를 처리한다.
- 신한 전월 실적 미충족 시 마일 가치가 0이다.
- 신한 추가 마일은 기존 월간 사용액을 반영하고 월 2,000마일을 넘지 않는다.
- 신한 기본 마일은 추가 마일 한도 소진 후에도 계속 계산된다.
- 10·15·20원 마일 가치에 따라 유효비용만 달라지고 현금성 비용은 유지된다.
- 선택적 Naver·신한 환율을 입력하면 해당 수단에만 적용된다.
- 유효비용 차이가 정확히 1%인 경우 근소한 차이 경고가 표시된다.
- 0, 음수, NaN, 무한대 입력이 거부되며 동일 비용 정렬이 결정적이다.

### 저장·PWA·UI 검증

- 새로고침과 오프라인 재실행 후 마지막 입력과 설정이 복원된다.
- 7일이 지난 추천 기록은 제거되고 최근 기록은 유지된다.
- JSON 내보내기와 전체 초기화가 저장 상태와 일치한다.
- 서비스 워커 업데이트 실패나 네트워크 단절 시 최근 입력값으로 계산할 수 있다.
- 320px, 375px, 390px 너비에서 가로 스크롤 없이 입력·결과·상세 내역을 사용할 수 있다.
- 키보드만으로 입력과 결과 상세를 조작할 수 있고 레이블, 오류, 상태 메시지가 보조기술에 전달된다.

## 10. 구현 순서와 커밋 단위

1. 프로젝트 스캐폴드, 테스트 환경, PWA 설정을 구성한다.
2. 타입·날짜 제한 규칙·순수 계산 엔진과 단위 테스트를 작성한다.
3. 입력 폼, 고급 설정, 순위 및 상세 내역 UI를 구현한다.
4. 버전형 로컬 저장, 7일 기록 정리, 내보내기·초기화를 구현한다.
5. 오프라인 동작과 모바일 접근성을 검증하고 GitHub Pages 배포 설정을 마친다.
6. 공식 조건을 재확인하고 QA 결과와 미확정 사항을 완료보고서에 남긴다.

각 단계는 독립적으로 빌드·테스트 가능한 상태에서 의미 있는 커밋으로 남긴다. 구현 세션에서 `code-team`을 명시적으로 사용할 경우 전용 브랜치와 해당 스킬의 planner→review→QA→eval 커밋 흐름을 따른다.

## 11. 완료 조건

- 네 결제수단이 동일 입력으로 비교되고 계산 내역과 가정이 화면에 노출된다.
- 근소한 차이, 오래된 환율, 날짜 만료 규칙, 근사 환율에 대한 경고가 작동한다.
- 계산 단위 테스트와 저장·PWA·모바일 시나리오가 통과한다.
- 정적 배포물에 API 키나 개인 데이터 전송 코드가 없다.
- 날짜 의존 조건의 공식 출처와 마지막 확인일을 사용자가 확인할 수 있다.
- 구현 및 운영 방법, 공식 프로모션 갱신 절차와 알려진 근사 한계가 문서화된다.

## 12. 확정 가정

- MVP 기준 여행일은 2026-07-01 이후이며 결제 통화는 JPY다.
- 환율 자동 조회는 구현하지 않고 사용자가 수동 입력한다.
- 마지막 환율과 입력 시각을 저장하며 24시간 이후 오래된 값으로 표시한다.
- 공통 환율을 기본으로 쓰고 Naver Pay와 신한카드는 필요할 때만 별도 보정한다.
- 잔여 혜택은 정밀 원장이 아니라 사용자가 갱신하는 대략값이다.
- 최근 추천 결과는 7일 보관하되 혜택 잔여액을 자동 차감하지 않는다.
- 유효비용 차이 1% 이내는 사실상 근소한 차이로 본다.
- 마일 가치는 10·15·20원 프리셋을 제공하며 기본값은 15원이다.
